# Aether 后续迭代路线图

## 1. 文档范围

本文只记录后续建议迭代，不再展开已完成的一期闭环能力。后续规划重点放在复用效率、质量判断、资产治理、生成服务适配和可视化管理。

图片拆解为可组合素材的完整方案见 `visual-asset-memory-design.md`。

## 2. 迭代原则

- 优先增强现有数据资产的可检索、可比较、可演化能力。
- 避免过早绑定单一图片生成服务或单一审美标准。
- 所有自动判断都应保留解释和用户确认点。
- 新能力应尽量复用现有 visual asset、prompt record、generation run 和 asset 结构。
- 本地个人使用体验优先，团队协作和常驻服务放到后续阶段。

## 3. Phase 2: 风格资产引擎

Phase 2 的目标是让风格资产从“可保存”升级为“可搜索、可比较、可维护”。

### 3.1 风格向量化与多维评分

为每个风格生成可计算的维度向量，而不是只依赖结构化文本字段。

建议维度：

- 画法媒介
- 色彩倾向
- 光影语言
- 构图模式
- 情绪强度
- 材质纹理
- 镜头语言
- 细节密度
- 后期效果

用途：

- 相似风格召回
- 风格聚类
- 风格地图
- Prompt 推荐
- 视觉 review 阈值校准

### 3.2 风格搜索与过滤

为风格库增加更强的搜索能力。

建议支持：

- 按名称、标签、摘要全文搜索
- 按 style profile 字段过滤
- 按最近使用、最近生成、用户喜欢率排序
- 按参考图数量、生成成功率、review 通过率筛选
- 按相似风格推荐

### 3.3 风格版本治理

后续可以扩展完整版本治理能力。

建议能力：

- 风格变更历史
- 风格分支树
- 合并记录和合并原因
- 版本 diff
- 回滚到旧版本
- 标记稳定版、实验版、废弃版

不要把相似风格自动合并成不可逆操作。合并、归档、覆盖仍应由用户确认。

### 3.4 风格圣经

为成熟风格生成更完整的设计系统文档。

建议内容：

- 色彩规则
- 光影规则
- 构图规则
- 材质规则
- 推荐主体类型
- 禁止元素
- 适配比例
- 适配生成 Skill
- 示例 prompt
- 视觉 review 常见偏差

风格圣经适合在某个风格已有多张参考图和多次生成反馈后生成，不建议对草稿风格自动生成。

## 4. Phase 3: 生成质量闭环

Phase 3 的目标是让生成结果不只是被保存，而是持续反哺风格和 Prompt 质量。

### 4.1 Review 阈值校准

基于历史 review 数据校准风格一致性阈值。

建议能力：

- 按风格统计 review 通过率
- 统计常见偏差类型
- 根据用户最终喜欢/不喜欢反向修正 review 阈值
- 区分“风格偏差”和“用户偏好偏差”
- 为不同风格设置不同的一致性阈值

### 4.2 自动修订建议

当视觉 review 判断为 `minor_deviation` 或 `major_deviation` 时，生成更可执行的修订建议。

建议输出：

- 应强化的风格词
- 应削弱或移除的词
- 建议调整的比例、构图或质量参数
- 可直接用于下一次生成的 revised prompt
- 是否建议重新生成

这里仍不应自动重生，除非用户显式要求自动迭代。

### 4.3 多候选生成与对比

支持同一个 prompt 一次生成多张候选，并进行横向比较。

建议能力：

- 候选图网格展示
- 每张图独立 visual review
- 自动排序：风格一致性、Prompt 对齐、用户偏好
- 用户选择最佳结果
- 将失败候选的偏差记录进统计

### 4.4 Prompt 学习

基于用户反馈和 review 结果，分析哪些 prompt 片段有效、哪些片段容易造成偏差。

建议能力：

- 高贡献风格词识别
- 低效或负面 prompt 词识别
- 风格模板自动建议
- negative prompt 优化建议
- 按生成 Skill 维护不同 prompt 偏好

## 5. Phase 4: 多生成 Skill 适配

Phase 4 的目标是让 Aether 能在不同图片生成 Skill 之间保持稳定输出质量。

### 5.1 Skill 参数映射

不同生成 Skill 对比例、质量、种子、风格强度等参数的命名可能不同。

建议建立参数映射层：

```json
{
  "aspectRatio": {
    "imagegen": "aspectRatio",
    "rightcodes-imagegen": "aspectRatio",
    "custom-skill": "ratio"
  }
}
```

### 5.2 Prompt 方言适配

同一个风格描述在不同生成服务中的最佳写法可能不同。

建议能力：

- 通用 refined prompt
- 针对 Skill 的 provider prompt
- 参数兼容性检查
- 不支持参数的降级说明
- provider 返回错误的可读化解释

### 5.3 Provider 效果档案

记录不同生成 Skill 在不同风格上的表现。

建议统计：

- 成功率
- 平均 review 分
- 常见失败原因
- 常见风格偏差
- 用户喜欢率
- 平均耗时和成本

## 6. Phase 5: 资产治理

Phase 5 的目标是让本地资产库长期可维护。

### 6.1 资产去重与清理

随着生成图片持续积累，后续需要治理资产体积。

建议能力：

- 基于 sha256 的重复资产识别
- 未引用资产扫描
- 失败生成的临时文件清理
- 大文件统计
- 按素材、日期、生成 Skill 清理
- 导出前完整性检查

### 6.2 导入、导出与备份

建议提供项目级迁移能力。

导出内容：

- SQLite 数据
- visual assets
- prompt records
- generation runs
- reference assets
- generated assets
- config snapshot

导入时需要处理：

- 路径重写
- asset hash 冲突
- style id 冲突
- schema 版本迁移

### 6.3 多项目资产隔离

当用户同时维护多个风格库时，需要清晰隔离。

建议能力：

- 多 workspace 配置
- 项目级 style library
- 跨项目风格复制
- 全局风格收藏
- 项目间资产引用检查

## 7. Phase 6: 可视化界面

Phase 6 的目标是降低风格库管理和生成历史回看成本。

### 7.1 本地 Web UI

建议先做按需启动的本地 UI，而不是常驻服务。

核心页面：

- 风格库
- 风格详情
- 参考图浏览
- Prompt 记录
- 生成历史
- visual review 结果
- 资产统计

### 7.2 风格地图

基于风格向量和标签构建可视化地图。

用途：

- 查看风格聚类
- 发现重复风格
- 找到相近风格
- 观察风格演化关系

### 7.3 生成对比工作台

用于比较同一 prompt 或同一风格下的多次结果。

建议能力：

- 多图并排
- 参数对比
- Prompt diff
- review diff
- 用户评分
- 一键保存最佳版本为新参考图或风格分支

## 8. Phase 7: 团队与协作

这一阶段不建议过早投入，除非个人本地工作流已经稳定。

建议能力：

- 团队共享风格库
- 权限控制
- 审核流
- 风格命名规范
- 变更审批
- 远端同步
- 冲突解决

## 9. 风险与边界

### 9.1 自动评分不应替代用户判断

视觉 review、审美评分和 Prompt 学习都只能作为建议。最终是否保留、重生、合并或分支，应由用户确认。

### 9.2 不要把内容偏差误判为风格偏差

风格 review 应关注画法、色彩、光影、构图、材质、后期等可迁移特征，不应因为主体内容变化就判定风格失败。

### 9.3 不要过早做复杂 RAG

在风格数量不大时，SQLite、结构化字段和轻量相似度足够。向量库和复杂 RAG 应在风格规模、搜索需求、延迟需求明确后再引入。

### 9.4 Provider 适配要保持松耦合

Aether 应记录生成 Skill、参数、输出和 review 结果，但不应把核心数据结构绑定到某个 provider 的私有字段。

### 9.5 资产归档需要可解释

自动归档生成图后，仍要保留 `original_output`、hash、mime、size 等信息，方便追踪来源和排查重复资产。

## 10. 建议优先级

### P0: 下一步最值得做

- 风格搜索与过滤
- 生成历史查询
- review 统计和偏差聚合
- 资产去重扫描
- 生成 Skill 参数映射

### P1: 稳定后推进

- 风格版本治理
- 多候选生成对比
- Prompt 学习
- 风格圣经
- 导入导出备份

### P2: 数据积累后推进

- 风格向量化
- 风格地图
- Provider 效果档案
- 本地 Web UI

### P3: 长期方向

- 团队共享风格库
- 审核流
- 远端同步
- 风格知识图谱
